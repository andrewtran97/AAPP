# Proprietary Technology Boundary

## 1. Purpose

This document defines the boundary between implemented repository behavior, target architecture, and proprietary candidate technology for Agent Black Box.

B29A is documentation-only.

This document does not claim legal exclusivity, patent protection, external certification, or external validation.

## 2. Current Implementation Boundary

The current repository contains a reference implementation.

Current implemented areas include:

- phase gates
- local validation scripts
- evidence-oriented records
- deterministic policy abstraction
- deterministic risk signals
- local evidence performance measurement
- documentation and examples

## 3. Proprietary Candidate Boundary

Agent Black Box may describe proprietary candidate technology.

Acceptable wording:

- proprietary candidate technology
- candidate control pattern
- target control architecture
- evidence-oriented control plane
- documentation boundary
- future implementation boundary

Unsupported wording must be avoided.

## 4. Candidate Technology Areas

Candidate areas include:

- Evidence-Control Protocol
- Agent Signal Contract
- Decision Boundary Ledger
- Governance Verdict Layer
- Judge Court Layer
- Circuit Pack
- future evidence pipeline optimization

These are candidates unless implemented, tested, validated, and accepted in a scoped phase.

## 5. Evidence-Control Protocol Boundary

The Evidence-Control Protocol describes a target chain:

scope -> intent -> policy -> identity -> execution boundary -> state ledger -> evidence -> verification -> governance -> incident closure -> receipt

Current repository behavior must not be overstated beyond accepted phases.

## 6. Agent Signal Contract Boundary

The Agent Signal Contract is a target structured event contract between agents and control layers.

It may include:

- signal id
- scope id
- intent id
- actor id
- tool id
- resource id
- policy decision id
- evidence digest
- governance state
- issue time

B29A does not add schema implementation.

## 7. Decision Boundary Ledger Boundary

The Decision Boundary Ledger is a target record of why a decision happened, what evidence supports it, and which authority path applied.

It may record:

- actor
- action
- resource
- decision
- reason codes
- policy digest
- approval requirement
- evidence link

B29A does not add ledger implementation.

## 8. Governance Verdict Layer Boundary

The Governance Verdict Layer is a target layer for classification, redaction, retention, rights, export, and training-use decisions.

Canonical verdict examples may include:

- ALLOWED
- REDACTED
- BLOCKED
- MALFORMED
- UNSAFE
- UNSUPPORTED
- RETENTION_VIOLATION
- EXPORT_NOT_ALLOWED
- TRAINING_NOT_ALLOWED
- TENANT_BOUNDARY_VIOLATION

B29A does not add governance runtime implementation.

## 9. Judge Court Layer Boundary

The Judge Court Layer is a review and ranking layer.

It must not be described as final authority for:

- merge
- deploy
- allow
- deny
- security-sensitive execution
- compliance acceptance

Final authority remains deterministic gates, CI, policy, and human or configured authority.

## 10. Legal Boundary

This repository must not claim legal protection unless a separate legal artifact exists.

Examples of required legal artifacts may include:

- patent filing
- trademark filing
- copyright registration where applicable
- assignment record
- license record
- external legal opinion

Without those artifacts, use candidate wording only.

## 11. External Validation Boundary

This repository must not claim external validation unless a separate validation artifact exists.

Examples of required validation artifacts may include:

- signed external review
- audit report
- formal certification record
- independent benchmark report
- accepted compliance mapping

Without those artifacts, use internal or target wording only.

## 12. Performance Boundary

Performance work must be tied to measured data.

Future performance candidates may include:

- batching
- digest optimization
- canonicalization optimization
- lower-level evidence buffer work
- async verification
- service split

A performance candidate must not become an implementation decision without measurement.

## 13. Supply Chain Boundary

Future supply-chain work may include provenance, signing, transparency logs, update safety, and BOM export.

B29A does not add supply-chain implementation.

Future supply-chain terms must not be described as current compliance.

## 14. Non-Goals

B29A does not add:

- patent claim
- trademark claim
- external audit claim
- external certification claim
- production deployment claim
- legal exclusivity claim
- new runtime behavior
- new policy behavior
- new scanner behavior
- new evidence pipeline behavior
- B30 external witness receipt implementation

## 15. Failure Modes

This boundary fails when:

- candidate technology is described as legally exclusive without artifacts
- future architecture is described as implemented behavior
- internal validation is described as external validation
- target performance work is described as measured improvement without benchmark data
- LLM review is described as authority
- compliance terms are used without proof artifacts

## 16. Current Phase Boundary

B29A records proprietary technology boundaries only.

Implementation belongs to later scoped phases.
