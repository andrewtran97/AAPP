# AAPP Claim Boundaries

AAPP / Agent Black Box is an AI Agent Control + Evidence Plane.

## Claim Boundary Table

| Claimed | Not Claimed |
|---|---|
| Local evidence artifacts for review. | External approval. |
| Deterministic policy paths in accepted phases. | Complete prevention of every misuse path. |
| Tamper-evident evidence artifacts. | Storage that cannot be altered outside system boundary. |
| Scope-gated local examples. | Unrestricted scanning authority. |
| Workload identity binding patterns. | Ownership of customer data. |
| Incident-ready casefile artifacts in accepted phases. | Final legal or regulator acceptance. |
| Developer-runnable local reference path. | Production deployment decision. |

## Public Wording Rules

Use:

- AI Agent Control + Evidence Plane
- deterministic policy path
- tamper-evident evidence
- scope-gated scan model
- workload identity binding
- evidence governance verdict
- incident-ready casefile
- audit-supporting artifact
- developer-runnable local reference path

Avoid:

- external approval claims
- overbroad security claims
- agency endorsement claims
- customer data ownership claims
- unsupported production-readiness claims
- language implying all attacks or misuse paths are stopped

## Evidence Boundary

Evidence artifacts support review. They do not replace legal, security, compliance, procurement, or deployment review.

## Data Boundary

AAPP must not claim ownership of customer data. No rights means no training use.

## Scope Boundary

Examples are local developer examples. Active scanning requires scope. Third-party targets require written scope.

## Runtime Boundary

B27C does not change runtime, scanner, or policy engine behavior. B27C does not implement B28.

## Review Boundary

External acceptance depends on reviewer requirements, environment, deployment controls, legal basis, and customer policy.
